# pdn-bbs тАФ design

**What:** a ground-up packet-radio BBS for the pdn node platform: personal mail + bulletins + NTS-shaped traffic, the classic terse W0RLI/FBB-style RF command surface, **webmail**, and **full LinBPQ-mail forwarding compatibility** (FBB B1F compressed forwarding as the core; B2F as the follow-on). Companion spec: [`linbpq-mail-compat.md`](linbpq-mail-compat.md) тАФ every wire-level requirement, sourced; its ┬з8 MUST line is this project's compat contract and its ┬з9 is the oracle checklist.

**Rules of the road** (Tom, 2026-06-11): own repo; **public interfaces only** тАФ the BBS reaches pdn exclusively through RHPv2 (the network plane any app gets), its web UI rides the app-gateway identity contract, and it ships as a standard `pdn-app.yaml` package. pdn contains zero BBS-specific code. Webmail is a product MUST (compat-spec ┬з8 parks it as forwarding-irrelevant, which it is тАФ it's still required here).

## Architecture

```
pdn-bbs.sln
  src/Bbs.Fbb        the wire: lzhuf (N=2048, CRC16+len32 "e1" container), SID build/parse,
                     FA/FC proposals + F> checksum, FS parse/emit, SOH/STX/EOT framing,
                     R:-line codec, the forwarding session FSM (caller + answerer roles),
                     MBL/RLI text fallback (SHOULD). No I/O тАФ pure codecs + an FSM over
                     an abstract duplex byte stream. Golden-vector-pinned.
  src/Bbs.Core       domain: SQLite message store (messages, recipients, BID dedup store
                     with lifetime, users/home-BBS, partner config, WP later), the
                     hierarchical-address parser/normaliser (WW-rooted), routing
                     (TO/AT/HR per-partner, implied-AT, best-HR-depth single-copy for P),
                     housekeeping (kill-by-age per type/state, BID lifetime, K-purge).
                     TimeProvider-driven throughout.
  src/Bbs.Console    the terse RF user surface over an abstract session stream:
                     prompt "de <CALL>>", ?/H A B X V N I L-family R/RM S-family K/KM OP
                     NODE, /ex + Ctrl-Z entry, paging, the exact acceptance shapes
                     (compat spec ┬з1).
  src/Bbs.Host       composition: RHPv2 client (rhp2lib-net if consumable, else a minimal
                     in-repo client to the pinned wire) binding the BBS callsign тАФ
                     inbound accept тЖТ Console session OR Fbb answerer (sniff the first
                     line: a SID = forwarding partner); outbound forwarding scheduler
                     (per-partner interval/immediate + connect scripts over RHP open);
                     ASP.NET loopback webmail (gateway identity headers, no second auth);
                     YAML config; pdn-app.yaml manifest (service + ui blocks).
  tests/             unit per project + golden vectors; tests/Bbs.Interop.Tests vs a
                     BPQMail-enabled LinBPQ container (compat spec ┬з7) тАФ the oracle lane.
  docker/            the oracle compose: linbpq + linmail.cfg (generated per spec ┬з7) +
                     netsim, mirroring packet.net's interop stack patterns.
```

Dependency rule: Fbb and Core never reference Host; Console references Core only; Host references all three. Nothing references packet.net source тАФ only published packages (and only if the RHP client lib is consumable; otherwise self-contained).

## The home-BBS requirement (Tom, 2026-06-11)

Although rare, a user MUST be able to adopt an instance as their **home BBS** without being the node owner тАФ e.g. a legacy TNC2 user across town using this node as their mail provider. Their address becomes `<call> @ <bbscall>.<HA>`; the world routes mail here and it waits in their inbox. This decomposes to:

1. **Local delivery beats forwarding** тАФ the load-bearing rule: a personal whose AT is us (our callsign, or an address under our own HA) gets **zero forward targets**, full stop; and a personal with no AT whose TO is a known local user stays local rather than matching any partner's wildcard AT. The dangerous failure is the silent one: a wildcard default route swallowing local users' mail. Pinned by tests before any wildcard-AT partner config goes live. **Implemented** (feat/local-delivery-rule): `RoutingEngine.Route` pre-empts the personal single-copy/flood dispatch with a `ResolvesToLocal` check (`Callsigns.BaseEquals(atBbs, _ownCall)` or `_ownHa.AreaContains(at)`; or no-AT + `RoutingRequest.ToIsLocalUser`) and returns an empty, `IsLocal`-flagged `RoutingDecision`; `RoutingService` sets `ToIsLocalUser` from `BbsStore.UserExists(<to>)` (base-callsign match). This restores LinBPQ's own "already here" local-first behaviour ([BPQ-SRC CheckAndSend]) the faithful single-copy port otherwise lacked. **Pinned by tests**: the leak test (`LocalDeliveryRoutingTests.NoAt_LocalUser_NeverLeaksToWildcardPartner` тАФ verified red before implementation) plus the engine-level `RoutingEngineTests` "local delivery beats forwarding" suite (at-is-us / under-HA / no-over-suppression / explicit-remote-AT-still-forwards / bulletin+traffic regression guards) and `BbsStoreTests.UserExists_тАж`.
2. **Auto-create the user on first inbound personal** тАФ mail can arrive before its owner ever connects; a skeletal user record makes it listable on their first `L`. **Implemented** (feat/auto-create-homed-user): the inbound delivery path (`InboundMessageReceiver.Deliver` тАФ inbound forwarding only; webmail compose / console sends never reach it) calls `AutoCreateHomedUser` after storing. The trigger is exactly the homed-personal case: the delivery is **type Personal** AND its **AT resolves to us** тАФ `RoutingEngine.AtResolvesToLocal(at)`, the own-call/own-HA half of rule #1's local-delivery signal (`Callsigns.BaseEquals(atBbs, _ownCall)` or `_ownHa.AreaContains(at)`; an empty AT is never us by this test, so the no-AT-existing-user case is excluded) AND the **TO is not already a known local user** (`BbsStore.UserExists`) тЖТ `BbsStore.EnsureUser(<to>)` inserts a skeletal row keyed by the base callsign (callsign only; name/QTH/Home left for the console's first-connect persistence). `EnsureUser` is idempotent (a no-op when the user already exists, matched on the base call), so repeat inbounds never duplicate or error. Explicit-remote-AT personals fail the AT-is-us test (they forward onward per rule #1), and bulletins / NTS-traffic are excluded by the Personal-only guard тАФ none auto-create. **Pinned by tests** (written first; the positive auto-create cases verified red on the pre-rule receiver, then green): `BbsStoreTests.EnsureUser_CreatesSkeletalRecord_IdempotentOnBaseCall` (store half: skeletal create, idempotence, base-call keying) and `AutoCreateHomedUserTests` (host half through the real `Deliver`: AT=own-call тЖТ user created + mail listable; AT under our HA тЖТ created; second inbound for the same TO тЖТ still one user; no-AT existing-user тЖТ delivered, no new user; explicit remote AT тЖТ no auto-create; bulletin + NTS addressed locally тЖТ no auto-create).
3. **WP announcements** (spec SHOULD, promoted by this requirement) тАФ emitting White Pages updates for homed users is how the rest of the network learns to route `<call> @ here`.
4. The RF console is already the full management surface (read/kill/send, `Home`/name/QTH persistence) тАФ no node-owner involvement needed; callsign-is-identity per network norms.

## The plain-language mandate (Tom, 2026-06-11 тАФ supersedes the compat-spec ┬з8 "classic user surface" MUST)

**All of the arcane knowledge goes: L/R/K/S, status letters, hierarchical-route incantations, the lot.** Interop compatibility is a *wire* property (FBB forwarding with partner BBSes тАФ untouched); it was never a reason to make humans speak W0RLI. The console becomes a plain-language line-mode surface that a TNC2 user on a dumb terminal can drive without folklore:

- **Canonical commands are words**: `help`, `list` (new mail), `read <n>`, `send <call>`, `reply`, `delete <n>`, `bulletins`, `topics`, `name`, `home`, `quit`. Single letters survive only as *accidental abbreviations* тАФ any unambiguous prefix of a word works (`l`, `r 3`, `q`), so a die-hard's fingers still work, but nothing requires the folklore and `help` explains everything in sentences.
- **Listings are sentences, not column dumps with status letters**: "3 new messages тАФ 1) from G4ABC, 12 Jun: Antenna party тАж", paclen-friendly lines, paged with a plain "more? (yes/no)".
- **Addressing is just a callsign**: `send g4abc` тАФ the system resolves where G4ABC lives via the network directory; the power form `send g4abc@gb7bsk` exists; regions/hierarchies NEVER surface to users.
- **Onboarding in sentences**: the first-connect flow asks for a name and offers "make this your home mailbox?" тАФ no Z/QTH/Home command trivia.
- **Sysop surface likewise**: `forwarding` (status), `queue`, `route <call>` (explain) тАФ see forwarding.md's vocabulary table.
- **Classic mode is a per-user preference** (Tom): the byte-exact W0RLI surface the W3 wave built (193 tests) is kept whole as `interface: classic` in the per-user settings тАФ for users whose automated clients (Winpack-era) pattern-match the legacy prompts. **Plain is the default**; a user flips it once with the `classic` command (typable by hand from any terminal, even a TNC2), from webmail, or the sysop sets it for them; the session engine picks the surface by callsign at connect (the caller is known before the greeting). Partner BBS forwarding is unaffected either way (SID-triggered, wire-level).
- So wave **U-1** builds the plain surface as the default alongside the kept classic implementation тАФ same session engine (lifecycle, store wiring, paging, persistence), two command tables/renderers, one preference. Runnable in parallel with F-1.

**U-1 тАФ plain-language console тЬЕ built (2026-06-12, feat/plain-language-console).** The plain surface is now the default, alongside the kept-whole classic surface, on one session engine. The shape:

- **`InterfaceMode` (Plain default | Classic)** is a nullable field on `UserSettings` (null тЖТ the config default, `BbsConsoleConfig.DefaultInterfaceMode = Plain` per the mandate); it persists automatically through `JsonUserSettingsStore` (the enum JSON round-trip is pinned by `JsonUserSettingsStoreTests`). `BbsConsoleSession` resolves the surface at connect from `settings.InterfaceMode ?? config.DefaultInterfaceMode` and routes greeting / prompt / dispatch to it; the classic path is byte-identical (its 193 tests stay green untouched). A BBS-flagged caller (automated peer pattern-matching the legacy prompts) is always served classic regardless of preference тАФ the plain default is for humans.
- **Plain command table** (word тЖТ operation, each mapping onto the SAME store ops the classic verbs use тАФ the surface is re-skinned, mail behaviour is not reimplemented): `help`, `list` (new mail) / `list <n>` (recent), `read <n>` (or bare `read` = unread mail), `send <call>` (power form `send call@bbs`), `reply <n>`, `forward <n> to <call>`, `delete <n>`, `bulletins`, `topics`, `name`/`qth`/`zip`/`home`/`info`/`delivered`/`pagelength`/`expert`, `classic`/`plain`, `quit`, `node`.
- **Prefix rule:** an exact word wins; otherwise any single word the typed prefix begins resolves it, several тЗТ a friendly "Did you mean тАж?", none тЗТ "I don't know тАж". A small muscle-memory shortcut map rescues the mandate's examples that the bare rule would call ambiguous тАФ `l`тЖТlist, `r`тЖТread, `re`тЖТreply, `q`тЖТquit, `h`тЖТhelp, `s`тЖТsend, `d`тЖТdelete, `b`тЖТbulletins, `n`тЖТname.
- **Sentence listings** ("You have 3 new messages:" then "\* 1) from G4ABC, 12 Jun: Antenna party"), no status letters/columns, an unread "\*" marker, paclen-trimmed subjects, paged with a plain `more? (yes/no)`. **Plain help** explains every command in full sentences. **Onboarding** asks the name and offers the home mailbox in sentences.
- **Mode toggle persists:** the plain `classic` command (and the classic `plain` command тАФ added to the classic dispatch without changing any existing verb's bytes) flips `InterfaceMode`, saves it, confirms in plain English, and takes effect from the next prompt; the choice sticks across reconnect via the JSON store (pinned: `Classic_PersistsAcrossReconnect`).

**U-1 follow-ons тЬЕ built (2026-06-12, feat/plain-sysop-ops-and-webmail-toggle):**
- **Plain sysop ops verbs** (`forwarding` / `queue` / `route <call>` тАФ forwarding.md's vocabulary ┬з58 + route phrasing ┬з77). Three read-only diagnostics on the PLAIN surface only (built jargon-free from the start; the kept-whole classic surface is untouched), sysop-gated at dispatch тАФ a non-sysop naming one gets a plain "that's a sysop-only command". `forwarding` shows each partner (dialled/off, dial cadence, queue depth via `GetForwardQueue(p).Count`); `queue` shows the pending forward queue in sentences; `route <call>`/`route <call>@<region>` builds a hypothetical personal, asks the SAME `RoutingEngine` the live host uses (`BbsConsoleConfig.HRoute` now threads our H-Route through), and renders the decision in plain words ("stays here", "carries mail for stations in region gbr.euro (closest match)", catch-all, "nowhere to go") тАФ never depth/HR jargon. The sysop verbs match exactly for everyone (so the refusal fires) but only join the PREFIX pool for a sysop, so a user's `forwar` stays an unambiguous `forward` (no leak) and `r`=read survives. **Named deferrals (visible):** the forwarding *action* verbs (forward-now / re-route) trigger real dials тЖТ a later slice; and live per-partner *health* (last cycle / failure / next retry тАФ forwarding.md wart 3) isn't surfaced because the `ForwardingScheduler` keeps it in a private loop variable, so `forwarding` names it as deferred rather than faking it.
- **Webmail interface-mode toggle тЬЕ built.** `WebmailOptions` gained `IUserSettingsStore Settings` (wired from `HostComposition` тАФ the same singleton the console session uses); a `GET /settings` page shows the user's current plain/classic mode (plain default for a never-set user) with a radio control, and `POST /settings` loads the user's `UserSettings`, sets `InterfaceMode` and `Save`s it (other settings preserved) тАФ so a webmail flip is the persisted choice the next console connect reads. Gated by `WithCallsign`/`FindCallsign` like every page; prefix-correct under the `X-Forwarded-Prefix` mount; a nav link added. (Just the interface-mode toggle; the fuller webmail sysop forwarding/BID view тАФ forwarding.md wart 3/8/10 тАФ remains the bigger F-2.)

## Load-bearing decisions

1. **Session demux тАФ greet immediately** (F-0; supersedes the silent first-line peek): one RHP-bound callsign serves both users and partners, and the BBS speaks FIRST, like LinBPQ itself (the silent peek deadlocked against a real LinBPQ caller, which waits for our text before sending anything). On accept the demux instantly sends our SID line, then starts the Console session so its greeting/prompt flows at once тАФ but holds the console's INPUT behind a first-line gate while the demux peeks the first inbound line. A forwarding opener (`[тАж-тАж]` SID shape, or `;FW:` per spec ┬з1.1) тЖТ the console is aborted via the gate (it has consumed no input, so a partner's SID can never be eaten by the new-user name prompt) and awaited so its writes flush, then the Fbb answerer runs in continue-mode (`FbbSessionConfig.SidAlreadySent`: the FSM starts at SID-parse; the peeked bytes are fed in). Anything else тЖТ the gate opens and the line is the console's first command. A silent caller sees the greeting while the demux waits `demuxFirstLineWaitSeconds`; at expiry the session is the console's and input flows normally thereafter (a later SID is just an invalid command). For a known partner the console skips the banner (the BBS flag, spec ┬з1.1), so the wire is the classic `[SID]` + `de CALL>` transcript.
2. **lzhuf**: implement from the compat spec ┬з6 byte layouts, pinned by the research golden vectors (`"Hello World!\r\n"` тЖТ `b5 66 0e 00 00 00 ea 7c ...`; the ascmail corpus vector) and round-trip property tests. XMODEM CRC16 over len+bitstream, little-endian, prepended. N=2048/F=60/THRESHOLD=2/fill 0x20, MSB-first 16-bit accumulators.
3. **Forwarding FSM** is sans-IO: `Advance(line|block) тЖТ actions`. Both roles in one machine, tested against scripted transcripts (including the wl2k-go/JNOS/BPQ asymmetries: BPQ `F>` always checksummed vs JNOS bare for FA; FS `H` = defer when talking, accept-tolerant when listening; STX len-0 = 256 inbound, тЙд250 emitted; the 6-byte restart preamble parse-tolerated).
4. **Identity/trust for webmail**: the gateway's `X-Pdn-Gateway: 1` + `X-Pdn-User` headers are the auth boundary (loopback bind, per the app-gateway contract). BBS users map pdn usernames тЖФ callsigns via its own user table.
5. **SID**: advertise `[PDN-<ver>-B1FHM$]` minimum (never containing "BPQ" тАФ the compat spec's gating trap); add `2` per-partner when B2F lands. **B2F landed** (feat/b2f-wire): per-partner `allowB2` (default off тЗТ B1 unchanged) drives the `2` digit; B2 is active for a session iff `allowB2` тИй the partner's SID `B2`. Outbound proposes uniform `FC EM` + ships a B2 object through the same B1 container/framing; inbound accepts `FC` (same dup-BID/size/hold checks as `FA`), decodes the B2 object and stores it through the identical store/route path (so the home-BBS rules #1/#2 + the R-chain loop/age guard apply to B2 inbound too). The answerer advertises `2` only to a partner record we set `allowB2` on, so we never accept `FC` from a non-`allowB2` caller (the original `-` guard, intact). **B2 completeness тЬЕ shipped (2026-06-12):** multi-recipient (`To:`/`Cc:`) + `File:` attachments are now fully stored, relayed and downloadable тАФ `DeliverFc` stores all To/Cc + attachments, `BuildB2Object` emits them, webmail renders Cc + per-attachment download links; the store gained a `recipients.cc` flag + an `attachments` table (additive migration). The single-To/no-attachment wire is byte-identical (GB7RDGтЖФpdn untouched). **Send-side file slice тЬЕ shipped (2026-06-12)** (feat/compose-file-upload): webmail compose now uploads a local file with a per-send choice тАФ 7plus-encode it (universal; the parts ride in the body text and reach any path, reassembled inbound by a pdn recipient's `SevenPlusAssembler`) or send it as a binary B2 attachment (rides the attachments plumbing тЖТ `BuildB2Object` `File:` part; B2-capable partners only). `POST /compose` reads `multipart/form-data` (`form.Files` + the text fields); a `MaxUploadBytes` ceiling (default 256 KiB) rejects oversize cleanly; uploaded names are path-stripped. The 7plus parts are appended verbatim (NOT through the CR body discipline тАФ `ReplaceLineEndings` would corrupt code-alphabet byte 0x85 as U+0085 NEL; pinned by a byte-exact encodeтЖТstoreтЖТscanтЖТassemble round-trip test). This closes the earlier "webmail compose upload" deferral. **One named deferral remains** from B2 completeness: per-*home* fan-out (a recipient with a different `@home` is stored but the message forwards once on the primary route with the full To: list тАФ FBB re-distributes; no per-home copies). The send-side slice itself defers (named): per-part bulletins for a large 7plus file (MVP embeds all parts in one body) and pretty Sent-view rendering of an outgoing 7plus message. See forwarding.md "B2F negotiation" for all of these + the oracle finding (real LinBPQ strips its own To + drops Cc but preserves the File: attachment; its user-entry surface can't originate multi-To/attachment).
6. **Storage**: one SQLite db in the package state dir (`PDN_APP_STATE`), resilient-open pattern, schema versioned.
7. **Oracle-first**: every forwarding behaviour lands with (a) a transcript test from the spec, then (b) an assertion against the live LinBPQ container before it's called done тАФ same diff-oracle discipline that paid off in RHPv2 R-4/R-5. Tom's m0lte/linbpq `tests/integration/fbb_partner.py` + `bpqmail_cfg.py` are the proven harness to port/reuse.
8. **Connect scripts (F-0, spec ┬з4.4)**: full script semantics over the RHP bearer. An optional FIRST `C [port] <target>` names the RHP open (the port rides the open; digi paths are warned-unsupported); every later line тАФ including a second `C`, which at a node prompt is the remote node's own connect command тАФ is sent verbatim with a response wait between lines (progress = ` CONNECTED` / `OK` / a `>` prompt; the ┬з4.4 failure list тАФ BUSY/FAILURE/SORRY/тАж тАФ fails the cycle into the backoff-retry path); `PAUSE n` delays; the other ┬з4.4 directives are recognised (kept off the wire) and warned. After the last line the script layer itself "waits for a SID or `>`" (┬з4.4 verbatim), consuming node chatter (CTEXT, `*** Connected to тАж` progress lines) so the Fbb caller FSM sees a clean stream from the SID on. Each wait is bounded by the partner's ConTimeout (named deviations: per-wait rather than whole-handshake; case-insensitive scans). Stepless plans (bare `connect:` / none) skip the script layer entirely тАФ the FSM waits out the SID, exactly the pre-F-0 behaviour. Failures carry the attempt transcript (what we sent / what came back / where it stopped).
